CONTEXTO DE SESION - paquete "banda" (fabrica_ws)
==================================================
Fecha: 2026-07-11

RESUMEN
-------
Se dejó el paquete ROS2 "banda" (src/banda) funcionando de punta a punta:
xacro -> RViz -> Gazebo Classic, con una banda transportadora estática y
una caja de prueba que cae sobre ella. Todo lo de abajo ya está aplicado
en el repo, esto es solo un registro de QUÉ se cambió y POR QUÉ, y qué
queda pendiente.


ARCHIVOS TOCADOS / CREADOS
---------------------------
src/banda/xacro/banda_transportadora.xacro
    - Meshes: "meshes/x.stl" -> "package://banda/meshes/x.stl"
      (ROS2 no soporta $(find pkg) de ROS1; package:// es el equivalente
      que entiende RViz/robot_state_publisher).
    - Se agregó el link "base_footprint" (sin geometría/inercia) como
      nuevo root del URDF, con joint fixed hacia body_link en
      xyz="0 0 0.728". Así, al spawnear en Gazebo (que coloca el root en
      z=0), el robot completo (incluidas las patas) queda sobre el
      suelo en vez de atravesarlo.

src/banda/xacro/gazebo.xacro
    - Se agregó <static>true</static> en el <gazebo> del modelo.
      Motivo: con gz sdf -p se confirmó que TODOS los links se fusionan
      en un solo cuerpo rígido (fixed joints -> lump), pero al ser
      dinámico, el contacto entre las mallas STL cóncavas de las patas
      de soporte (support_L/support_R) y el plano del suelo era
      numéricamente inestable en ODE -> el modelo temblaba/vibraba.
      Al marcarlo "static", el motor de física ya no lo trata como
      cuerpo dinámico (no puede temblar), pero sigue generando
      colisión/fricción normal contra objetos dinámicos apoyados
      encima (la caja de prueba, futuros objetos reales sobre la banda).
    - El plugin "libROS2ConveyorBeltPlugin.so" sigue referenciado ahí
      pero AUN NO ESTÁ INSTALADO en el sistema (ver pendientes).

src/banda/urdf/test_box.urdf (nuevo)
    - Caja simple 8x8x8 cm, 0.15 kg, geometría de caja primitiva (sin
      mesh), fricción mu1=mu2=1.0. Se usa solo para probar la banda.

src/banda/launch/display.launch.py (nuevo)
    - robot_state_publisher + joint_state_publisher (o _gui con
      use_joint_state_gui:=true) + RViz. Sin Gazebo. Para revisar rápido
      el modelo/TFs.

src/banda/launch/gazebo.launch.py (nuevo)
    - Incluye gazebo_ros/launch/gazebo.launch.py (Gazebo Classic),
      publica robot_description (con ParameterValue(..., value_type=str)
      para que robot_state_publisher no intente parsear el URDF como
      YAML), hace spawn_entity del robot, RViz opcional (use_rviz,
      default true), y 6s después spawnea test_box.urdf sobre el
      origen de la banda (spawn_test_box:=false para desactivarla).
    - SetEnvironmentVariable('ROS_LOCALHOST_ONLY', '1'):
      en WSL2 el descubrimiento DDS por multicast fallaba
      intermitentemente y spawn_entity.py nunca detectaba el servicio
      /spawn_entity aunque existiera. Forzar localhost lo arregla.
    - SetEnvironmentVariable('GAZEBO_MODEL_PATH', ...) apuntando al
      directorio padre de share/banda: Gazebo Classic NO entiende el
      esquema "package://" (solo "model://"), pero sdformat trata
      package:// igual que model:// si el directorio padre de "banda"
      está en GAZEBO_MODEL_PATH. Sin esto, el modelo se sincronizaba en
      gzclient pero sin geometría visible.
    - Altura de spawn de test_box (z≈0.82): calculada leyendo el
      bounding box real de meshes/band.stl (top de la banda en
      z=+0.003 respecto a band_link) + el offset de base_footprint
      (0.728) + margen de caída.

src/banda/rviz/banda.rviz (nuevo)
    - Grid + RobotModel + TF, Fixed Frame = base_footprint.

src/banda/setup.py
    - Se agregó instalación de launch/*.launch.py, xacro/*.xacro,
      meshes/*.stl, rviz/*.rviz, urdf/*.urdf a share/banda/... (con
      ament_python esto NO se instala solo, hay que declararlo).

src/banda/package.xml
    - exec_depend agregados: xacro, robot_state_publisher,
      joint_state_publisher, joint_state_publisher_gui, rviz2,
      gazebo_ros, gazebo_plugins.

Limpieza: se borraron 3 archivos basura "*.xacro:Zone.Identifier" en la
raíz del paquete (metadatos de Windows sin contenido útil).


FIXES A NIVEL DE SISTEMA (fuera del repo, ya aplicados por el usuario)
-----------------------------------------------------------------------
- sudo ip link set lo multicast on
  El loopback de WSL2 no traía la bandera MULTICAST, lo que rompía la
  sincronización interna de Gazebo Transport entre gzserver y gzclient
  (el modelo se creaba en la física pero nunca se dibujaba en el
  cliente). Se agregó a /etc/wsl.conf:
      [boot]
      command="ip link set lo multicast on"
  para que persista en cada arranque de WSL (requiere "wsl --shutdown"
  una vez desde PowerShell para tomar efecto).


CÓMO PROBAR
------------
cd ~/fabrica_ws
colcon build --packages-select banda
source install/setup.bash
ros2 launch banda display.launch.py     # solo RViz
ros2 launch banda gazebo.launch.py      # Gazebo + RViz + caja de prueba
    # args opcionales: use_rviz:=false  spawn_test_box:=false


PENDIENTE / PRÓXIMOS PASOS
----------------------------
1. El plugin "libROS2ConveyorBeltPlugin.so" (referenciado en
   gazebo.xacro) NO está instalado todavía. Por eso la caja cae y se
   queda quieta sobre la banda, pero no se desliza. Para que la banda
   realmente "mueva" objetos hay que:
     - Clonar y compilar en este workspace el paquete de
       IFRA_ConveyorBelt / ros2_conveyorbelt:
       https://github.com/IFRA-Cranfield/IFRA_ConveyorBelt
     - Revisar que el plugin apunte al link correcto: el plugin de
       referencia espera un link llamado "belt"; puede que haya que
       renombrar band_link o ajustar el código fuente del plugin
       (Load()/GetLink).
     - Una vez compilado, la velocidad se activa/regula llamando al
       servicio ROS2 "/CONVEYORPOWER"
       (conveyorbelt_msgs/srv/ConveyorBeltControl, power = 0..100).
   OJO: ahora que el modelo es "static", hay que confirmar que el
   plugin de banda (cuando se instale) siga funcionando con un link
   estático; si el plugin depende de mover/leer velocidad del link
   band_link como cuerpo dinámico, puede necesitar ajuste. La mayoría
   de estos plugins funcionan vía contactos (aplican velocidad
   directamente al objeto que toca la banda), así que static debería
   ser compatible, pero no se pudo probar en esta sesión por falta del
   plugin.

2. Nada más queda pendiente del flujo básico xacro/RViz/Gazebo: RViz
   confirmado por el usuario ("funciona perfecto"), Gazebo confirmado
   sin temblor tras el fix de static (pendiente de que el usuario lo
   confirme visualmente tras el último cambio).
